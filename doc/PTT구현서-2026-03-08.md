# PTT 구현서 (PTT Implementation Specification)

**프로젝트**: light-livechat SFU Server  
**작성일**: 2026-03-08  
**버전**: v0.5.1  
**범위**: PTT 미디어 파이프라인 서버 구현 (Phase E-0 ~ E-4)

---

## 1. 개요

### 1.1 목표

WebRTC SFU 위에서 무전기(PTT, Push-To-Talk) 모드를 구현한다.
여러 화자가 하나의 채널에서 교대로 발화하며, 수신 측에서는
**하나의 연속된 오디오/비디오 스트림**으로 인식해야 한다.

### 1.2 핵심 과제

일반 화상회의(Conference)에서는 각 참가자의 SSRC가 독립적이다.
PTT에서는 화자가 바뀔 때마다 서버가 RTP 헤더를 리라이팅하여
수신 측 Jitter Buffer가 단일 SSRC 스트림으로 인식하게 해야 한다.
이는 선택이 아닌 **본질적 요구사항**이다.

### 1.3 설계 원칙

- **Conference 모드 zero impact**: 모든 PTT 분기는 `room.mode == Ptt` 가드로 보호
- **light의 정체성 유지**: SSRC 리라이팅 최소 범위 (RTP 헤더 4필드 + PLI + NACK)
- **MBCP 용어 차용, 프로토콜은 독자**: 3GPP TS 24.380 용어와 상태 전이를 차용하되, WebSocket JSON 시그널링 사용

## 2. 시그널링 (v0.5.0에서 완성)

### 2.1 Floor Control 상태 머신

```
                  Floor Request
    ┌──────┐    (speaker 없음)     ┌──────┐
    │ IDLE │ ──────────────────→   │TAKEN │
    │      │ ←──────────────────   │      │
    └──────┘   Floor Release /     └──────┘
               Revoke / Leave          │
                                       │ Floor Request (다른 사용자)
                                       ▼
                                    DENIED
```

### 2.2 시그널링 Opcode

| 방향 | Op | Name | 설명 |
|------|-----|------|------|
| C→S | 40 | FLOOR_REQUEST | PTT 누름 (발화권 요청) |
| C→S | 41 | FLOOR_RELEASE | PTT 뗌 (발화권 반납) |
| C→S | 42 | FLOOR_PING | 발화자 생존 확인 (2초 주기) |
| S→C | 141 | FLOOR_TAKEN | 누군가 발화권 획득 (브로드캐스트) |
| S→C | 142 | FLOOR_IDLE | 채널 비어있음 (브로드캐스트) |
| S→C | 143 | FLOOR_REVOKE | 서버 강제 회수 (발화자에게) |

### 2.3 타이머

| 타이머 | 값 | 설명 |
|--------|------|------|
| T2 (max burst) | 30초 | 최대 연속 발화 시간, 초과 시 Revoke |
| T_FLOOR_PING | 2초 | 클라이언트 PING 전송 주기 |
| T_FLOOR_TIMEOUT | 5초 | PING 미수신 시 Revoke |
| Floor Timer Task | 2초 | 서버 타이머 체크 주기 (T2 + PING 감시) |

```

│                                    ├─ floor.request(A) → Granted    │

│    ├─ [4] Video Rewrite (PT=96):                                   │

│    │    │   ├─ is_vp8_keyframe? → YES → pending=false, rewrite     │

│    │    └─ pending == false → rewrite                               │
│    └─ [5] fan-out → subscriber egress queues                       │
│                                                                     │
│  Publisher B (not floor holder)                                     │
│    │ RTP                                                            │
│    ▼                                                                │
│  [1] ingress.handle_srtp()                                         │
│    ├─ decrypt_rtp                                                   │
```

### 3.2 SSRC 리라이팅 연산 (오프셋 기반 상대 연산)

화자 전환 시점에 서버가 기록하는 상태:

| 값 | 설명 |
|----|------|
| `origin_base_seq` | 새 화자의 첫 원본 시퀀스 |
| `virtual_base_seq` | 가상 시퀀스 연속값 (이전 화자 마지막 + 1) |
| `origin_base_ts` | 새 화자의 첫 원본 타임스탬프 |
| `virtual_base_ts` | 가상 타임스탬프 연속값 (이전 화자 마지막 + guard gap) |

연산 공식:
```
virtual_seq = original_seq - origin_base_seq + virtual_base_seq
virtual_ts  = original_ts  - origin_base_ts  + virtual_base_ts
```

Guard gap (화자 전환 시 ts 간격):
- Audio: 960 (Opus 48kHz × 20ms)
- Video: 3000 (VP8 90kHz × ~33ms)

**비순차 도착에 안전**: UDP 특성상 패킷 순서가 바뀌어도 원본 스트림의 상대적 순서 관계가 그대로 보존된다. `seq++` 카운터 방식과의 결정적 차이.

### 3.3 VP8 키프레임 감지 (RFC 7741)

```
RTP Payload:
  [VP8 Payload Descriptor]
    byte 0: X(extension) | R | N | S(start) | R | PID
    → S bit: 파티션 시작 여부
  [VP8 Payload Descriptor Extension] (X=1일 때)
    → I(PictureID) / L(TL0PICIDX) / T|K 건너뛰기
  [VP8 Bitstream]
    byte 0, bit 0: frame_type (0=key, 1=inter)
```

`is_vp8_keyframe()` 함수가 RTP 헤더 → payload descriptor → bitstream까지 파싱하여 I-frame을 감지한다.

### 3.4 주요 지점별 동작 요약

| 지점 | Conference 모드 | PTT 모드 |
|------|----------------|----------|
| **ingress gate** | 전원 통과 | floor holder만 통과, 나머지 DROP |
| **RTP cache** | 원본 plaintext 저장 | 동일 (리라이팅 전에 저장) |
| **Audio RTP** | 원본 그대로 fan-out | 가상 SSRC/seq/ts로 리라이팅 후 fan-out |
| **Video RTP** | 원본 그대로 fan-out | 키프레임 대기 → 가상 SSRC/seq/ts 리라이팅 |
| **SR relay** | 전원 릴레이 | floor holder의 SR만 릴레이 |
| **NACK** | 원본 SSRC/seq로 cache 조회 | 가상seq → 원본seq 역산 후 cache 조회 |
| **RR/PLI relay** | media_ssrc로 publisher 찾기 | 가상SSRC → 원본SSRC 변환 후 publisher 찾기 |
| **Egress task** | 변경 없음 | 변경 없음 (리라이팅은 ingress에서 완료) |
| **Floor Timer** | 없음 | 2초 주기 T2/PING 감시, 타임아웃 시 Revoke |

## 6. WebRTC 제약사항 및 중요 포인트

### 6.1 SSRC 리라이팅은 선택이 아닌 필수

WebRTC의 수신 측(Chrome libwebrtc)은 SSRC 기반으로 미디어 스트림을 식별한다.
화자가 바뀔 때 다른 SSRC의 RTP가 갑자기 도착하면:
- subscribe SDP에 해당 SSRC가 선언되어 있지 않으면 **Chrome이 드롭**
- 선언되어 있더라도 Jitter Buffer가 별도 스트림으로 처리 → **연속 재생 불가**

따라서 서버가 가상 SSRC로 통일하는 것이 PTT의 필수 조건이다.

### 6.2 키프레임 대기의 불가피성

화자 전환 시 새 publisher의 첫 RTP가 P-frame(inter frame)이면
수신 측 디코더가 참조 프레임 없이 디코딩 불가 → 영상 깨짐/멈춤.

서버가 PLI를 보내고 I-frame(키프레임)이 올 때까지 P-frame을 드롭하는 것은
**화상 무전기에서 피할 수 없는 지연**이다.

예상 지연: PLI 전송 → publisher encoder 반응 → 키프레임 생성 → 서버 수신
약 100~300ms (네트워크 + encoder 반응 시간).

### 6.3 오프셋 연산 vs 카운터 방식

단순 `virtual_seq++` 방식은 **절대 사용 불가**:
- UDP는 패킷 순서를 보장하지 않음
- 도착 순서대로 가상 seq를 찍으면 원본의 순서 관계가 파괴됨
- 수신 측 Jitter Buffer가 재정렬 불가 → 영상/음성 끊김

오프셋 기반 상대 연산은 원본 스트림의 상대적 순서를 100% 보존한다.

### 6.4 RTP Cache와 리라이팅 순서

```
decrypt → rtp_cache.store(원본) → rewrite(가상) → fan-out
```

RTP Cache에는 **원본 plaintext**를 저장해야 한다.
리라이팅된 패킷을 저장하면 NACK 역매핑이 불필요해지지만,
RTX 재전송 시 가상 SSRC/seq/ts가 변경되어야 하는 복잡도가 생긴다.
원본 저장 + NACK 역산이 더 단순하고 안전하다.

### 6.5 SDP와 가상 SSRC의 관계 (미해결)

현재 subscribe SDP(SdpBuilder)는 각 publisher의 **원본 SSRC**를 선언한다.
PTT 모드에서 실제로 수신하는 것은 **가상 SSRC**이다.

이 불일치가 Chrome에서 문제를 일으킬 수 있다:
- Chrome이 SDP에 없는 SSRC의 RTP를 드롭할 가능성
- 또는 SSRC demux 실패 → 잘못된 트랙에 매핑

**해결 방안** (클라이언트 SDK 구현 시):
1. PTT 모드에서 subscribe SDP에 가상 SSRC를 선언
2. 또는 ROOM_JOIN 응답에 가상 SSRC 정보를 포함하여 SdpBuilder가 사용
3. sdes:mid extmap 제거 상태이므로 SSRC 기반 demux가 작동 — 가상 SSRC 선언 필수

### 6.6 브라우저 Jitter Buffer의 SSRC 전환 처리

화자 전환 시 marker bit을 설정하는 것은 수신 측 Jitter Buffer에
"새 세그먼트 시작"을 알리는 힌트다.

그러나 이것은 **RFC 상의 권고**이지 강제가 아니며,
브라우저 구현에 따라 동작이 다를 수 있다:
- Chrome: marker bit → Jitter Buffer 플러시, 비교적 잘 처리
- Safari/Firefox: 검증 필요

### 6.7 Opus DTX와 묵음 구간

Opus DTX(Discontinuous Transmission) 활성 시,
묵음 구간에서 comfort noise 패킷만 간헐적으로 전송된다 (2~3 pps).

PTT 모드에서 비발화자는 gate에서 드롭되므로
DTX 여부와 무관하게 subscriber에게 도달하지 않는다.
발화자의 DTX는 정상 작동하며 리라이팅도 정상 적용된다.
